# AI-Kubernetes-Upgrades

A community project for **safe, auditable Kubernetes cluster upgrades** across on-premises (kubeadm), Amazon EKS, Google GKE, and Azure AKS.

---

## What This Project Does

Given a source and target Kubernetes version, this project:

1. **Assesses upgrade feasibility** — produces a comprehensive risk report identifying everything that could break during or after the upgrade (deprecated APIs, CRD incompatibilities, controller/operator risks, webhook issues, networking, storage, security, runtime).  
   → Output: `cluster-upgrade-feasibility-and-risks.md`

2. **Generates a platform-specific execution plan** — a step-by-step upgrade runbook tailored to whether the cluster is on-prem (heavy: kubeadm, etcd, SSH, manual drains) or managed (light: EKS/GKE/AKS provider CLI commands).  
   → Output: `plan.md`

## File Structure

```
├── prompt.md                                # Template — feasibility assessment prompt
├── cluster-upgrade-execution-plan.md        # Template — execution plan generator
├── LICENSE                                  # Apache License 2.0
├── README.md                                # This file
│
│   ↓ (generated by running the prompts) ↓
│
├── cluster-upgrade-feasibility-and-risks.md # OUTPUT — the upgrade risk assessment report
└── plan.md                                  # OUTPUT — the platform-specific execution plan
```

**Templates vs Outputs:** `prompt.md` and `cluster-upgrade-execution-plan.md` are immutable templates that are never overwritten. The generated reports go into separate files.

## Is It Safe to Run?

**Yes — `prompt.md` is 100% read-only.** Every command inside it uses read operations only:

| Tool | Operations Used |
|------|----------------|
| `kubectl` | `get`, `version`, `api-resources`, `cluster-info`, `top` |
| `kubeadm` | `version` (no `upgrade`, `init`, `apply`) |
| `etcdctl` | `version` (no `snapshot save`, `defrag`) |
| `crictl` / `containerd` / `docker` | `--version` |
| `aws eks` | `describe-cluster`, `describe-addon`, `list-addons`, `describe-nodegroup`, `list-nodegroups` |
| `gcloud container` | `clusters describe`, `node-pools list`, `operations list`, `get-server-config` |
| `az aks` | `show`, `get-upgrades`, `nodepool list`, `nodepool show` |
| `kubent` | Reads cluster resources + Helm v3 Secrets/ConfigMaps (no mutations) |
| `pluto` | Reads cluster resources + Helm release Secrets (no mutations) |

**No `apply`, `create`, `delete`, `patch`, `edit`, `scale`, `drain`, `cordon`, or `upgrade` commands are present.** The assessment only reads cluster state — it never mutates it.

> ⚠️ **Note:** `cluster-upgrade-execution-plan.md` (the plan template) and its output `plan.md` DO contain destructive commands (cordon, drain, upgrade, etc.). These are strictly execution instructions for the operator, not automated scripts. Review thoroughly before executing.

## Platform Support

| Platform | Assessment | Execution Plan |
|----------|-----------|----------------|
| On-Prem (kubeadm) | Full | Heavy — manual etcd backup, kubeadm upgrade, SSH drain per node, cert checks |
| Amazon EKS | Full + EKS-specific add-ons | Light — `aws eks` CLI, managed node groups, add-on version sync |
| Google GKE | Full + release channels | Light — `gcloud` CLI, node pool upgrade (SURGE/RECREATE) |
| Azure AKS | Full + CNI/CSI checks | Light — `az aks` CLI, system→user pool ordering, max-surge |

Managed platforms (EKS/GKE/AKS) require significantly fewer steps than on-prem because the cloud provider handles etcd, control plane HA, certificates, and kubelet distribution.

## How to Use

### Step 1 — Run the assessment

Provide `prompt.md` as a system prompt to an AI assistant (like opencode) along with:

```text
SOURCE_VERSION=1.27
TARGET_VERSION=1.29
```

The assistant will guide you through running the kubectl/provider commands against your cluster and produce `cluster-upgrade-feasibility-and-risks.md`.

### Step 2 — Generate the execution plan

Feed `cluster-upgrade-execution-plan.md` as a prompt, with `cluster-upgrade-feasibility-and-risks.md` as context. The assistant will detect the platform, extract all findings, and produce `plan.md` — a runbook specific to your cluster.

### Step 3 — Execute the plan

Review `plan.md` carefully. Execute the steps during a planned maintenance window. Managed cluster upgrades typically complete in 30–90 minutes; on-prem upgrades scale linearly with node count.

## Requirements

- `kubectl` configured with cluster access
- Platform CLI (`aws`, `gcloud`, or `az`) if using a managed cluster
- `jq` for JSON parsing (optional but recommended)
- `kubent` (Kube No Trouble) for API deprecation scanning — `brew install kubent`
- `pluto` (Fairwinds) for Helm-aware deprecation scanning — `brew install pluto`
- Read access to cluster resources (no admin/write permissions needed for the assessment) + read access to Secrets (for Helm release history scanning)
